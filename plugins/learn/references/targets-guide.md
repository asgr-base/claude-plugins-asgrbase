# 改善対象別 書き込みガイドライン

`/learn` スキルが改善提案を生成する際の「何を書くべきか・避けるべきか」のガイドライン。

---

## CLAUDE.md

### 追記すべき内容

- **操作ルール**: 特定ツール・サービスの失敗パターンと回避策
- **必須確認事項**: 作業開始前に毎回確認すべきこと（認証チェック等）
- **禁止事項**: 過去に問題が発生した操作・アプローチ
- **ツール別ルール**: ツールやサービス固有の注意点

### 避けるべき内容

- 1回しか起きていない偶発的なエラー
- 標準的なベストプラクティス（自明な内容）
- コードパターン・アーキテクチャ（コードで表現すべき）
- 長い説明文（1-2行で書けない内容は別ファイルへ）

### 挿入位置のルール

- 既存セクションに関連する内容 → そのセクションの末尾に追記
- 新しいカテゴリの内容 → 「## HOW（必須ルール）」の「操作ルール」セクションに追加
- CLAUDE.md を肥大化させない: 追記1件=削除1件 を意識する

### 書き方の基準

```markdown
# Good（具体的・行動可能）
GitHub API レスポンスの JSON 解析は制御文字エラーが頻発する。Python直接パースではなく `jq` を使用すること

# Bad（曖昧・自明）
GitHub API を使うときは注意すること
```

## .claude/rules/*.md

> **読み込み特性**: セッション開始時に**必ず**読み込まれる。内容が増えるほどコンテキストを常に消費する。

### 書くべき内容の基準

- **短く行動可能なルール**（1-3行が理想）
- 毎回のセッションで守る必要があるもの
- 「してはいけない」「必ず〜すること」という禁止・義務の明示

長い説明が必要な場合は「ルールだけ rules/、詳細は references/」に分離する:
```
# rules/tools-debug.md に書く（常時読み込み）
GitHub API の JSON 解析は `jq` を使うこと。詳細: .claude/references/api-patterns.md

# references/api-patterns.md に書く（遅延読み込み）
## GitHub API パターン集
[詳細な説明・コード例・トラブルシューティング]
```

### 新規ファイル作成の判断基準

以下をすべて満たす場合に新規ファイルを作成する:
- 特定のツール・ドメイン・ワークフローに特化したルール群
- 5行以上のルールが生まれる見込みがある
- CLAUDE.md の既存セクションでは対応できない

### 既存ファイルへの追記

- `tools-debug.md`: ツールエラー・接続失敗・フォールバック手順
- `markdown-style.md`: Markdown記法・フロントマター・出典記載に関するルール
- `spec-index.md`: 仕様書・ドキュメントの参照先追加

### 避けるべき内容

- CLAUDE.md に既に書かれている内容の重複
- 1回限りの例外的な処理
- 長い説明文（→ references/ に分離する）

## .claude/references/*.md

> **読み込み特性**: 明示的に参照された時のみ読み込まれる（遅延読み込み）。  
> **重要**: 参照ポインタがなければ永遠に読まれない。追加する場合は必ず参照元とセットにする。

### 参照ポインタのセット原則

`references/` に新規ファイルを追加・追記を提案する場合、必ず以下もセットで提案する:

| 参照元の候補 | 追加方法 |
|------------|---------|
| 関連スキルの SKILL.md | フェーズのSTEPに `[references/xxx.md](references/xxx.md) を参照` を追加 |
| `rules/` の関連ファイル | ルールの末尾に `詳細: .claude/references/xxx.md` を追記 |
| CLAUDE.md の関連セクション | セクション末尾に `詳細: .claude/references/xxx.md` を追記 |

### 書くべき内容の基準

- **長い・詳細な説明**（3行以上になるもの）
- 特定のタスク・スキル実行時にだけ必要なナレッジ
- 繰り返し調査が必要になる外部情報（URLとサマリー）
- 判断基準・チェックリスト（複数回使われるもの）

### 新規ファイル作成の判断基準

- 既存ナレッジファイルのカバー範囲外
- 複数の異なるセッションで参照されそうな内容
- **参照元が明確に存在する**（参照元がないなら Memory を検討）

## .claude/skills/*/SKILL.md

### 既存スキルへの追記 vs 新スキル作成

| 状況 | 判断 |
|------|------|
| 既存スキルの手順が不足・誤り | → 既存スキルを更新 |
| 同じ操作を **3回以上** 手動で繰り返した | → 新スキル作成を提案 |
| 既存スキルのスコープ外の操作 | → 新スキル作成を提案 |

### 追記フォーマット（既存スキルへの追記）

```markdown
### トラブルシューティング: [問題名]
**症状**: [どういう状態になるか]
**原因**: [なぜ発生するか]
**解決策**: [具体的な手順]
```

### 新スキル提案の出力形式

スキルを実際に作成するのではなく、要件を整理して提案するに留める:
```markdown
**新スキル案**: [スキル名]
**目的**: [何を自動化するか]
**トリガー**: [いつ使うか]
**主な手順**: [3-5ステップ]
→ 実際の作成は `skill-creator` スキルで行ってください
```

## Hooks

### 自動化の候補パターン

| フック種別 | 自動化できる操作の例 |
|-----------|-------------------|
| **PreToolUse** | 特定ツール実行前の認証チェック、パス検証 |
| **PostToolUse** | ファイル編集後の構文検証、Lint実行 |
| **Stop** | セッション終了前のサマリー生成、ログ保存 |

### 提案する際の注意

- hookの追加はユーザーの通常作業フローに影響する → 高確信度の場合のみ提案
- 既存hookとの競合がないか確認する
- hookスクリプトは `.claude/hooks/scripts/` に配置する慣習

---

## 確信度スコアの判断基準

| 確信度 | 基準 | 提案上限 |
|--------|------|---------|
| **高** | セッション内で2回以上発生、または重大な時間損失 | 無制限 |
| **中** | 1回発生、次回また同じ問題が起きそう | 3件まで |
| **低** | 細かい改善・表現変更 | 1件まで |
